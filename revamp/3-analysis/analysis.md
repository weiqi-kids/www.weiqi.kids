## 1. 受眾深度分析

### 1.1 主要受眾 A：二代接班型（Persona A）

#### Persona 摘要

| 項目 | 描述 |
|------|------|
| 角色 | 中小企業二代或新任主管 |
| 年齡/背景 | 32–45 歲，已在自家或前東家累積 5–15 年產業經驗，新近承擔策略決策 |
| 目標 | 把家族／既有業務「升級」為 AI 時代的版本，但不想自建 50 人技術團隊 |
| 痛點 | (1) 內部沒有技術人才；(2) 外包不知道找誰；(3) 商會多半在交換業務，沒人陪走 AI 落地 |
| 行為特徵 | 主動 Google「中小企業 AI 導入」「產業 AI 應用」；參加少量精選社團；願意付時間不付高額會費 |

#### Jobs-to-be-Done（[假設]）

| 任務類型 | 具體任務 |
|----------|----------|
| 功能性 | 找到能幫我把「產業 know-how」變成 AI 工具的合作對象；最好已經有具體案例可看 |
| 情感性 | 不想被當成「不懂 AI 的中小企業主」；想要平等對話、被視為知識貢獻者而非客戶 |
| 社會性 | 對外能說「我們公司有導入 AI」；對內能跟父親／前輩證明轉型有具體成果 |

#### 使用者旅程（[假設]）

| 階段 | 行為 | 問題／疑慮 | 需要的資訊 | 期望的體驗 |
|------|------|-----------|------------|------------|
| 意識 | Google「中小企業 AI 導入」/ 朋友轉貼 weiqi.kids 連結 | 這是什麼組織？跟我有關嗎？ | 一句話定位、有沒有跟我同產業的成員 | 首屏看到「中小企業＋AI」相關訊息 |
| 考慮 | 看會員、看案例、看專案 | 這群人是真的做事還是純社交？我去能做什麼？ | 11 位會員實績卡、合作案例敘事、具體 AI 工具列表 | 5 分鐘內能判斷「值得花時間了解」 |
| 決策 | 寫信／報名活動 | 寫信會不會被當行銷漏接？費用多少？ | 明確的 mailto + 收件後流程說明、明確「不收費」 | 一鍵 mailto，主旨自動填入、明確期待回信時程 |
| 使用 | 加入活動／提合作 | 我能貢獻什麼？怎麼開始？ | 入會流程、首次合作可走的「最小路徑」 | 不是被推銷，是被當作「跨域夥伴」 |
| 推薦 | 轉介給其他二代朋友 | 怎麼介紹？對方會不會被當客戶？ | 一句話定位、可轉貼的會員實績連結 | 有 OG 圖、有 share-friendly URL |

#### 資訊需求（[假設]）

| 想知道什麼 | 擔心什麼 | 需要的證據 |
|------------|----------|------------|
| 「11 位會員具體是誰？有沒有跟我同產業？」 | 怕加入後發現都是某一行業大宗，自己變局外人 | 11 位會員的本業＋產業標籤＋對應 AI 工具 |
| 「他們真的做出什麼？開源 = 噱頭嗎？」 | 怕「開源」只是用詞，實際 repo 沒在維護 | 40+ 專案的 GitHub stars / commits / 媒體引用 |
| 「我要付多少錢？」 | 怕隱藏會費／培訓費 | 首頁明確「完全免費、無會員費」 |
| 「我能貢獻什麼？是不是只有 CEO 等級才有資格？」 | 怕門檻高、被拒 | 「合作邀請」流程清楚、有最小參與路徑 |

| 痛點 | (1) 個人能力強但難找對等的跨域合作對象；(2) 不想當免費勞工，但也不需要再賺錢；(3) 對「AI」有興趣但缺工程同伴 |
| 行為特徵 | 已退出主流社交圈、偏好實體互動、對「圍棋」「茶會」「讀書會」這類場域有親近感 |

#### Jobs-to-be-Done（[假設]）

| 任務類型 | 具體任務 |
|----------|----------|
| 功能性 | 找到能把我「腦中的條文／臨床經驗／策略案例」轉成 AI 工具的工程同伴 |
| 情感性 | 想要被當作「知識提供者」而非「老人家」；保有專業尊嚴 |
| 社會性 | 對外能說「我在退休後做了有意義的事」；對家人能展示具體貢獻 |

#### 使用者旅程（[假設]）

| 階段 | 行為 | 問題／疑慮 | 需要的資訊 | 期望的體驗 |
|------|------|-----------|------------|------------|
| 意識 | 朋友推薦／圍棋圈耳語 | 這群人在幹嘛？是不是又是 NGO 募款？ | 「不收費、開源」明確訊息 | 看到具體成果不是抽象使命 |
| 考慮 | 看會員履歷、看活動紀錄 | 我能融入嗎？有沒有我這個世代的人？ | 11 位會員年齡分布、定期活動紀錄 | 看到 50+ 世代會員、有實體棋會 |
| 決策 | 想見面聊聊 | mailto 太冷漠，想要實體聚會 | **棋會邀請入口**（不是純 mailto）| 有「下次棋會時間／地點」資訊 |
| 使用 | 提供領域知識、參與案例 | 我的時間值得嗎？產出歸誰？ | CC-BY 4.0 授權、貢獻者具名 | 被當作 co-author 而非 free labor |
| 推薦 | 介紹專業圈同儕 | 對方一樣會擔心被當免費勞工 | 已有會員的「貢獻＋具名」具體例子 | 有 case study 可轉發 |

#### 資訊需求（[假設]）

| 想知道什麼 | 擔心什麼 | 需要的證據 |
|------------|----------|------------|
| 「實體場域在哪？多久一次？」 | 怕純線上、無法看人 | 定期棋會時間、地點、過往照片 |
| 「我的貢獻會被署名嗎？」 | 怕變成理事長一個人的成果 | 11 位會員實績卡＋對應 AI 工具明確掛名 |
| 「我能用我的步調投入嗎？」 | 怕被當業務員壓進度 | 入會無 KPI、無強制參與條款 |
| 「下一代如何延續？」 | 退出後成果是否仍可用 | CC-BY 4.0 公開授權聲明 |

### 1.3 次要受眾 C：AI 工程師／開源貢獻者

#### Persona 摘要（簡略）

| 項目 | 描述 |
|------|------|
| 角色 | 28–40 歲，技術背景，找開源協作或產業案例 |
| 行為 | 從 GitHub trending / X / Hacker News 路徑進入；先看 repo 再看官網 |
| 痛點 | 多數開源 AI 缺領域知識／真實使用者；想找「有產業 partner 的 repo」 |

#### Jobs-to-be-Done（[假設]）

| 任務類型 | 具體任務 |
|----------|----------|
| 功能性 | 找有真實使用者／領域知識的開源 AI 專案、可長期貢獻 |
| 情感性 | 想做「有用的東西」而非「練 side project」 |
| 社會性 | GitHub profile 加分；履歷有具體案例 |

#### 旅程關鍵差異

- **入口從 GitHub 而非官網** → repo README / Topics 比首頁文案重要
- **不會看會員介紹** → 看 commits / issues / contributing guide
- 對 weiqi.kids 主敘事「商界夥伴」**不感興趣**，但對「跨產業領域知識的真實 partner」有興趣

## 4. 機會識別

### 4.1 必須補的內容（競品有、我們沒有、受眾需要）

| 內容 | 為什麼必須 | 競品參考 |
|------|------------|----------|

### 4.2 差異化機會（競品沒有、受眾需要）

| 機會 | 為什麼是機會 | 如何執行 |

## 5. 關鍵發現摘要

### 最重要的 3 個發現

1. **G1+G5+G6 是「不做就完全卡住」的三件事**
   → Umami 注入（量測）+ Hero 重寫（定位對齊）+ mailto CTA（決策入口）。三者構成 P0a「敘事骨幹」，**必須在 4-strategy 階段把三者列為第一輪實作的最小集合**，不可拆散。

2. **G2+G3 是「11 位會員從數字變成人」的關鍵雙向連結**
   → 「會員實績卡」單做沒用，必須同時做「AI 工具頁 ↔ 會員頁」雙向連結；否則 Persona A 看會員不知道對應產出、看工具不知道誰主筆。**這是 P0b 內容生產的最高槓桿點**。

3. **Persona B（退休專業型）的決策依賴「實體場域」，與 Persona A（二代接班型）的決策依賴「數位轉換」不同**
   → 不能只做一條 mailto CTA 就以為涵蓋兩個 Persona。**需雙 CTA**：(a) mailto（適合 A）+ (b) 棋會邀請（適合 B）；這是 2-competitive §可借鏡做法 P0a 已點出，需在 `4-strategy` 明確排程。

### 改版重點方向

基於以上分析，建議 `4-strategy` 階段聚焦於：

1. **P0a「敘事骨幹」最小集合**（不可拆）：Umami 注入 → Hero 重寫 → mailto + 棋會邀請雙 CTA → 「不收費」USP 宣告 → Meta description 更新
2. **P0b「11 人實績」雙向連結**：11 位會員實績卡 ↔ 40+ AI 工具頁；先做 5 位高貢獻會員 + 對應 5 個旗艦專案的 deep link，其餘隨後補
3. **P1「差異化護城河」**：年度成果報告（YPO 範式）+ 議題化分類（3–4 個產業，依現有專案密度收斂）

> 本報告所有 Persona / JTBD / Journey 皆為 [假設]，建議用以下方式驗證：

| 假設 | 驗證方式 | 時程 |
|------|----------|------|
